DMA Splunk Export – Complete Analytics File Reference (v4.6.0)

Applies To: All three export scripts (Enterprise Bash, Cloud Bash, Cloud PowerShell) Version: 4.6.0 Last Updated: April 2026

This document is the authoritative reference for every analytics file generated by the DMA export scripts. It supersedes all previous versions. v4.6.0 eliminated all per-category detailed usage queries and replaced them with 6 global aggregate queries plus REST-based supplementary metadata.

Enabling Analytics Collection

Usage analytics are OFF by default (since v4.2.4). You must explicitly enable them:

# Enterprise
./dma-splunk-export.sh --usage ...

# Cloud (Bash)
./dma-splunk-cloud-export.sh --usage ...

# Cloud (PowerShell)
./dma-splunk-cloud-export.ps1 -Usage ...

Without --usage (or -Usage), the analytics queries documented below will not run.

Global Aggregate Queries (collect_app_analytics / Export-GlobalAnalytics)

These 6 queries run once globally (not per-app). Results are grouped by app so the DMA Server can split them into per-app data after import.

All queries support checkpoint resume – if a query completes successfully on a previous run, it will be skipped on subsequent runs (useful when re-running after partial failures).

Query 1: Dashboard Views

Output file: dashboard_views_global.json

Index: _audit

Timeout: 3600 seconds (1 hour)

SPL Query (all three scripts use identical logic):

search index=_audit sourcetype=audittrail action=search info=granted
    (provenance="UI:Dashboard:*" OR provenance="UI:dashboard:*")
    user!="splunk-system-user"
    earliest=-<USAGE_PERIOD>
| rex field=provenance "UI:[Dd]ashboard:(?<dashboard_name>[\w\-\.]+)"
| where isnotnull(dashboard_name)
| eval view_session=user."_".floor(_time/30)
| stats dc(view_session) as view_count, dc(user) as unique_users,
    values(user) as viewers, latest(_time) as last_viewed
    by app, dashboard_name
| sort -view_count

App scoping: When --scope-to-apps is active, an app filter (e.g., (app="app1" OR app="app2")) is injected into the base search.

How it works: - Filters _audit to only dashboard-triggered searches using the provenance field (format: UI:Dashboard:<name> or UI:dashboard:<name>) - Extracts the dashboard name from provenance using rex - De-duplicates views using a view session concept: user + floor(_time/30) groups all panel searches within a 30-second window as a single page load - dc(view_session) counts distinct page loads, not individual panel searches

Output fields:

Field Type Description
app string Splunk app the dashboard belongs to
dashboard_name string Dashboard XML/JSON filename (without extension)
view_count integer De-duplicated page load count
unique_users integer Number of distinct users who viewed the dashboard
viewers multivalue List of user IDs who viewed the dashboard
last_viewed epoch Timestamp of most recent view

Migration purpose: Prioritize which dashboards to migrate first based on actual usage. Dashboards with zero views in this data are candidates for archival (the DMA Curator computes never-viewed dashboards by diffing the dashboard list against this data).

v4.6.0 note: This replaces the old dashboard_views_top100.json. The provenance field is reliable; the old search_type=dashboard was NOT a native _audit field and produced unreliable results.

Query 2: User Activity

Output file: user_activity_global.json

Index: _audit

Timeout: 3600 seconds (1 hour)

SPL Query:

search index=_audit sourcetype=audittrail action=search info=granted
    user!="splunk-system-user" user!="nobody"
    earliest=-<USAGE_PERIOD>
| stats count as searches, dc(search_id) as unique_searches,
    latest(_time) as last_active
    by app, user
| sort -searches

App scoping: When --scope-to-apps is active, an app filter is injected into the base search.

How it works: - Counts every granted search per user per app - Excludes system users (splunk-system-user, nobody) - dc(search_id) gives the number of distinct searches (vs. count which includes re-runs)

Output fields:

Field Type Description
app string Splunk app context of the search
user string Splunk username
searches integer Total search executions
unique_searches integer Distinct search IDs
last_active epoch Most recent search timestamp

Migration purpose: Identify active users per app for training priorities, team planning, and UAT candidate selection. Users with zero activity are candidates for exclusion from Dynatrace access provisioning.

v4.6.0 note: This replaces the old per-app users_most_active.json, users_inactive.json, daily_active_users.json, and activity_by_role.json.

Query 3: Search Type Breakdown

Output file: search_patterns_global.json

Index: _audit

Timeout: 1800 seconds (30 minutes)

SPL Query:

search index=_audit sourcetype=audittrail action=search info=granted
    user!="splunk-system-user"
    earliest=-<USAGE_PERIOD>
| eval search_type=case(
    match(provenance, "^UI:[Dd]ashboard:"), "dashboard",
    match(search_id, "^(rt_)?scheduler__"), "scheduled",
    match(savedsearch_name, "^_ACCELERATE_"), "acceleration",
    match(search_id, "^SummaryDirector_"), "summarization",
    isnotnull(provenance) AND match(provenance, "^UI:"), "interactive",
    1=1, "other"
  )
| stats count as total_searches, dc(user) as unique_users,
    dc(search_id) as unique_searches
    by app, search_type
| sort -total_searches

App scoping: When --scope-to-apps is active, an app filter is injected into the base search.

How it works: - search_type is NOT a native _audit field. It is derived using the same logic as Splunk’s Monitoring Console (eval case()): - "dashboard" – provenance starts with UI:Dashboard: or UI:dashboard: - "scheduled" – search_id starts with scheduler__ or rt_scheduler__ - "acceleration" – savedsearch_name starts with _ACCELERATE_ - "summarization" – search_id starts with SummaryDirector_ - "interactive" – has a provenance starting with UI: but not dashboard - "other" – everything else

Output fields:

Field Type Description
app string Splunk app context
search_type string One of: dashboard, scheduled, acceleration, summarization, interactive, other
total_searches integer Total search count for this type
unique_users integer Distinct users running this type
unique_searches integer Distinct search IDs

Migration purpose: Understand the workload composition per app. A high scheduled count means more Dynatrace workflows will be needed. A high dashboard count means dashboard migration is the priority. A high interactive count means ad-hoc search capabilities (DQL notebooks) are important to users.

Query 4: Index Volume Summary

Output file: index_volume_summary.json

Index: _internal

Timeout: 600 seconds (10 minutes)

SPL Query:

search index=_internal source=*license_usage.log type=Usage
    earliest=-<USAGE_PERIOD>
| eval index_name=idx
| stats sum(b) as total_bytes, dc(st) as sourcetype_count,
    dc(h) as host_count, min(_time) as earliest_event,
    max(_time) as latest_event
    by index_name
| eval total_gb=round(total_bytes/1024/1024/1024, 2),
    daily_avg_gb=round(total_gb/<usage_days>, 2)
| sort -total_gb
| fields index_name, total_bytes, total_gb, daily_avg_gb,
    sourcetype_count, host_count, earliest_event, latest_event

Where <usage_days> is the numeric value of USAGE_PERIOD (e.g., 30 for 30d).

Skip behavior: When --skip-internal is set, writes:

{"skipped": true, "reason": "_internal index not accessible (--skip-internal). Use index metadata from REST API for size estimates."}

Output fields:

Field Type Description
index_name string Splunk index name (from idx field in license_usage.log)
total_bytes integer Total bytes ingested during the period
total_gb float Total gigabytes (rounded to 2 decimal places)
daily_avg_gb float Average daily ingestion in GB
sourcetype_count integer Number of distinct sourcetypes in this index
host_count integer Number of distinct hosts sending to this index
earliest_event epoch Earliest license event timestamp
latest_event epoch Latest license event timestamp

Migration purpose: Critical for Dynatrace Grail storage planning and cost estimation. The daily_avg_gb field directly maps to expected Grail ingestion volume.

Query 4b: Index Event Counts (Cloud Scripts Only)

Output file: index_event_counts_daily.json

Index: Any (uses | tstats which queries tsidx files directly)

Timeout: 60 seconds

Dispatch mode: Blocking (not async – this is a fast metadata query)

SPL Query:

| tstats count where index=* by index, _time span=1d
| rename index as index_name
| sort _time, -count

Availability: Cloud Bash and Cloud PowerShell only. The Enterprise script does not generate this file.

How it works: - | tstats reads tsidx metadata without scanning raw data, so it runs in seconds - Provides daily event counts per index as a fallback when _internal is not accessible - Runs as a best-effort query – errors are suppressed

Output fields:

Field Type Description
index_name string Splunk index name
_time epoch Day boundary timestamp
count integer Event count for that index on that day

Migration purpose: Volume trending and capacity planning. Useful when _internal license_usage.log is not accessible (common in Splunk Cloud).

Query 5: Alert Firing Stats

Output file: alert_firing_global.json

Index: _internal

Timeout: 3600 seconds (1 hour)

SPL Query:

search index=_internal sourcetype=scheduler
    earliest=-<USAGE_PERIOD>
| fields _time, app, savedsearch_name, status
| stats count as total_runs,
    sum(eval(if(status="success",1,0))) as successful,
    sum(eval(if(status="skipped",1,0))) as skipped,
    sum(eval(if(status!="success" AND status!="skipped",1,0))) as failed,
    latest(_time) as last_run
    by app, savedsearch_name
| sort -total_runs

App scoping: When --scope-to-apps is active, an app filter is injected into the base search.

Skip behavior: When --skip-internal is set, writes:

{"skipped": true, "reason": "_internal index not accessible (--skip-internal). Alert firing stats require scheduler logs."}

How it works: - Reads scheduler logs from _internal to determine which saved searches/alerts have been executing - | fields early in the pipeline limits memory usage on large environments - Breaks down executions into successful, skipped, and failed counts

Output fields:

Field Type Description
app string Splunk app the saved search belongs to
savedsearch_name string Name of the saved search or alert
total_runs integer Total scheduler executions
successful integer Executions with status=success
skipped integer Executions with status=skipped
failed integer All other statuses (failed, errored, etc.)
last_run epoch Most recent execution timestamp

Migration purpose: Identify which alerts are operationally critical (high run count + high success rate) vs. candidates for elimination (never run, always failing, or always skipped). The failed count helps identify broken alerts that should be fixed before or during migration.

v4.6.0 note: This replaces the old alerts_most_fired.json, alerts_with_actions.json, alerts_failed.json, alerts_never_fired.json, and alert_firing_trend.json.

Query 6: Alerts Inventory (Per-App)

Output file: <app_name>/splunk-analysis/alerts_inventory.json (one file per app)

Index: None (uses | rest command)

Timeout: 120 seconds per app

Dispatch mode: Blocking (not async – | rest queries are fast)

SPL Query (executed once per selected app):

| rest /servicesNS/-/<app>/saved/searches
| search (is_scheduled=1 OR alert.track=1)
| table title, cron_schedule, alert.severity, alert.track, actions, disabled
| rename title as alert_name

How it works: - Uses Splunk’s | rest command to query the saved searches REST endpoint scoped to each app - Filters to only scheduled searches (is_scheduled=1) or tracked alerts (alert.track=1) - This is the only per-app query remaining in v4.6.0 – it runs quickly because | rest is a metadata operation

Output fields:

Field Type Description
alert_name string Saved search / alert title
cron_schedule string Cron expression for the schedule
alert.severity string Alert severity level
alert.track string Whether alert tracking is enabled
actions string Configured alert actions (email, webhook, etc.)
disabled string Whether the alert is disabled

Migration purpose: Provides a per-app inventory of active alerts for migration planning. Combined with Query 5 (firing stats), this gives a complete picture of which alerts exist and which are actually running.

Supplementary Data (collect_usage_analytics / Export-UsageAnalytics)

These files are collected via REST API calls or | rest SPL commands. They do not require _audit or _internal access.

Ownership Mapping

Three files provide a complete ownership picture for user-centric migration planning.

dashboard_ownership.json

Location: dma_analytics/usage_analytics/dashboard_ownership.json

Collection method: - Cloud scripts: Direct REST API call to /servicesNS/-/-/data/ui/views (no search job) - Enterprise script: | rest SPL search job

Enterprise SPL Query:

| rest /servicesNS/-/-/data/ui/views
| table title, eai:acl.app, eai:acl.owner, eai:acl.sharing
| rename title as dashboard, eai:acl.app as app, eai:acl.owner as owner,
    eai:acl.sharing as sharing

App scoping (Enterprise only): When --scope-to-apps is active, an eai:acl.app filter clause is appended.

Output fields:

Field Type Description
dashboard string Dashboard title
app string Owning Splunk app
owner string Splunk user who owns the dashboard
sharing string Sharing level: user, app, or global

Migration purpose: Assign migration responsibility by owner. Map Splunk sharing levels to Dynatrace document permissions.

alert_ownership.json

Location: dma_analytics/usage_analytics/alert_ownership.json

Collection method: - Cloud scripts: Direct REST API call to /servicesNS/-/-/saved/searches (no search job) - Enterprise script: | rest SPL search job

Enterprise SPL Query:

| rest /servicesNS/-/-/saved/searches
| table title, eai:acl.app, eai:acl.owner, eai:acl.sharing, is_scheduled, alert.track
| rename title as alert_name, eai:acl.app as app, eai:acl.owner as owner,
    eai:acl.sharing as sharing

App scoping (Enterprise only): When --scope-to-apps is active, an eai:acl.app filter clause is appended.

Output fields:

Field Type Description
alert_name string Saved search / alert title
app string Owning Splunk app
owner string Splunk user who owns the alert
sharing string Sharing level: user, app, or global
is_scheduled string Whether the saved search is scheduled
alert.track string Whether alert tracking is enabled

Migration purpose: Ownership tracking for alerts. Combined with alert_firing_global.json, provides both “who owns it” and “is it actually running.”

ownership_summary.json

Location: dma_analytics/usage_analytics/ownership_summary.json

Collection method: - Cloud scripts: Computed from REST API data (no search job) - Enterprise script: | rest SPL search job with | append and | stats

Enterprise SPL Query:

| rest /servicesNS/-/-/data/ui/views
| stats count as dashboards by eai:acl.owner
| rename eai:acl.owner as owner
| append [
    | rest /servicesNS/-/-/saved/searches
    | stats count as alerts by eai:acl.owner
    | rename eai:acl.owner as owner
  ]
| stats sum(dashboards) as dashboards, sum(alerts) as alerts by owner
| sort - dashboards

Output fields:

Field Type Description
owner string Splunk username
dashboards integer Number of dashboards owned
alerts integer Number of saved searches / alerts owned

Migration purpose: Quick summary of content ownership across the environment. Identifies which users own the most content and should be primary stakeholders in migration planning.

REST Metadata

Three files provide supplementary metadata collected via direct REST API calls (no search jobs, no SPL).

saved_searches_all.json

Location: dma_analytics/usage_analytics/saved_searches_all.json

REST endpoint: GET /servicesNS/-/-/saved/searches

Query parameters:

output_mode=json
count=0
f=title
f=eai:acl
f=is_scheduled
f=disabled
f=cron_schedule
f=alert.track
f=alert.severity
f=actions
f=next_scheduled_time
f=dispatch.earliest_time
f=dispatch.latest_time

v4.5.1 fix: The f= parameters restrict the response to metadata fields only. Without f=, the endpoint returns ALL fields including the full raw SPL for every saved search, which can produce 256MB+ responses on large environments.

Post-processing (Cloud Bash only): When --scope-to-apps is active, results are filtered with jq to include only saved searches belonging to selected apps.

Content: Complete inventory of all saved searches with scheduling, alerting, and ACL metadata. Does not include the saved search SPL text (intentionally excluded via f= field selection to control response size).

Migration purpose: Provides a full inventory of saved searches for migration planning without requiring search job execution.

recent_searches.json

Location: dma_analytics/usage_analytics/recent_searches.json

REST endpoint: GET /services/search/jobs

Query parameters:

output_mode=json
count=100    (Cloud Bash)
count=1000   (Enterprise)

Content: List of recent/active search jobs with their metadata (SID, status, owner, app, etc.).

Migration purpose: Provides a snapshot of current Splunk workload at the time of export.

kvstore_stats.json

Location: dma_analytics/usage_analytics/kvstore_stats.json

REST endpoint:

GET /services/kvstore/status          (Cloud Bash, Cloud PowerShell)
GET /services/server/introspection/kvstore   (Enterprise -- uses introspection endpoint)

Query parameters:

output_mode=json

Content: KVStore service status and statistics.

Migration purpose: Identifies KVStore usage that may need to be migrated to Dynatrace (Grail tables, Settings objects, etc.).

Troubleshooting

Common Issues

Symptom Cause Resolution
Analytics files contain {"skipped": true, ...} --skip-internal was set or _internal is not accessible Expected behavior – use REST metadata for estimates
Dashboard views file is empty or has 0 results _audit index not accessible or no dashboard activity in the period Check permissions; try a longer --usage-period
Queries time out after 1 hour Very large _audit or _internal indexes Reduce --usage-period (e.g., 7d instead of 30d)
Checkpoint prevents re-run Previous run already collected this query Delete the checkpoint file in dma_analytics/.checkpoints/

Verifying Results

To manually verify a query, run the SPL in Splunk Web’s Search & Reporting app. Replace <USAGE_PERIOD> with your configured period (default: 30d for Enterprise, 7d for Cloud) and remove the ${app_filter} placeholder if not using app scoping.
